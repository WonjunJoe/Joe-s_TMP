02

디렉토리 구조

.omc/는 실행 중 상태를 파일별로 분리 저장한다

프로젝트별 상태가 3개 하위 디렉토리로 격리된다

  • project-memory.json — 프로젝트 메타데이터
  • sessions/{id}.json — 세션별 진행 스냅샷
  • state/mission-state.json — 현재 미션 추적
  • state/hud-state.json — HUD 상태
  • state/autopilot|ralph-state.json — 실행 모드별 상태

~/.claude/는 OMC 동작의 뼈대를 구성한다

  • CLAUDE.md — OMC 핵심 운영 가이드
  • settings.json — Claude Code 설정
  • .omc-config.json — OMC 프로젝트 설정
  • hud/omc-hud.mjs — 스태이터스라인 스크립트
  • plugins/ — 플러그인 레지스트리 및 캐시
03

핵심 설정 파일

CLAUDE.md는 모든 에이전트가 공유하는 운영 헌법이다

7개 섹션이 위임·라우팅·검증 규칙을 일원화한다

  • <operating_principles> — 위임, 검증, 경량화 원칙
  • <delegation_rules> — 직접 처리 vs 위임 기준
  • <model_routing> — Haiku/Sonnet/Opus 선택
  • <skills> — Tier-0 워크플로우 정의
  • <verification> — 완성 전 검증 프로토콜

두 설정 파일로 실행 모드·팀 크기·HUD를 제어한다

최소 설정으로 팀 에이전트 모드가 활성화된다

// .omc-config.json
{ "defaultExecutionMode": "ultrawork",
  "team": { "maxAgents": 3 } }

// settings.json
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" },
  "statusLine": { "command": "node $HOME/.claude/hud/omc-hud.mjs" } }
04

에이전트 시스템

복잡도에 따라 Haiku→Sonnet→Opus가 자동 배정된다

3단계 모델 라우팅이 비용 대비 품질을 최적화한다

⚡
Haiku
빠른 조회
단순 검색
🎯
Sonnet
표준 작업
구현, 분석
🏆
Opus
복잡한 작업
설계, 보안

29+ 에이전트가 5개 전문 레인으로 역할 분리된다

각 레인이 독립 책임을 가져 병렬 협업이 가능하다

레인에이전트모델
빌드/분석explore, planner, architect, executor, verifierHaiku~Opus
리뷰code/security/performance/api/style-reviewerSonnet~Opus
도메인test-engineer, designer, writer, qa-testerSonnet
제품product-manager, ux-researcher, analystSonnet
조율critic, visionOpus

고급 워크플로우가 전문 도메인 작업을 자동화한다

  • deep-interview — Socratic 심층 요구사항 인터뷰
  • deep-dive — trace + deep-interview 2단계
  • ultraqa — 테스트→검증→수정 QA 사이클
  • visual-verdict — 스크린샷 시각적 비교 QA
  • ai-slop-cleaner — AI 생성 코드 정리

07

상태 관리

상태가 3개 범위로 계층 분리되어 저장된다

프로젝트·전역·중앙화 3단계로 상태 유실을 방지한다

범위경로설명
프로젝트.omc/state/{name}.json현재 프로젝트 전용
전역~/.omc/state/{name}.json사용자 전역
중앙화$OMC_STATE_DIR/{id}/워크트리 삭제 후 유지

각 실행 모드는 독립 JSON 파일로 추적된다

6개 핵심 상태 파일이 미션·에이전트·모드를 각각 추적한다

파일역할주요 필드
mission-state현재 미션 추적agents[], status
hud-state스태이터스라인backgroundTasks[]
subagent-tracking에이전트 수명agents[], total
autopilot-stateautopilot 진행phase, goal, plan
ralph-stateralph 루프iteration, verified

원자적 쓰기·TTL·자동 마이그레이션으로 상태 일관성을 보장한다

  • 캐시 TTL 5초 — 읽기 성능 최적화
  • 자동 정리 4시간 — 오래된 상태 자동 삭제
  • 원자적 쓰기 — 부분 쓰기 방지
  • 자동 마이그레이션 — 레거시→표준 경로 변환

09

MCP / HUD / 매직 키워드 / 환경 변수

MCP 서버와 HUD가 실행 인프라를 제공한다

  • MCP 출력 정책 — strict 기본, 리다이렉트 지원
  • HUD 로드 순서 — 개발경로→캐시→npm→에러
  • 플러그인 — omc v4.9.1, skill-creator, mcp-server-dev

특정 단어 입력만으로 해당 스킬이 즉시 자동 실행된다

가장 긴 매치(구체적인 것)가 우선 적용된다

키워드트리거 스킬
"ralph", "don't stop"ralph
"autopilot", "build me"autopilot
"ulw", "ultrawork"ultrawork
"ralplan", "consensus"ralplan
"tdd", "deslop", "cancel"TDD / slop-cleaner / cancel

6개 환경변수로 OMC를 제어하며 DISABLE_OMC가 킬스위치다

환경변수만으로 병렬 실행·훅 스킵·전체 비활성화가 가능하다

변수기본값설명
OMC_STATE_DIR_(unset)_중앙화 상태 디렉토리
OMC_PARALLEL_EXECUTIONtrue병렬 실행 활성화
OMC_LSP_TIMEOUT_MS15000LSP 타임아웃
DISABLE_OMC_(unset)_OMC 전체 비활성화
OMC_SKIP_HOOKS_(unset)_특정 훅 스킵
10

사용 예시

6가지 핵심 시나리오로 OMC 활용법을 익힌다

  1. autopilot — 새 기능 완전 자동 구축
  2. ralph — 복잡한 버그 끝까지 수정
  3. ultrawork — 대규모 병렬 리팩토링
  4. ralplan — 아키텍처 설계 합의
  5. team — N개 에이전트 팀 조율
  6. deep-interview — 요구사항 심층 인터뷰

autopilot: 새 기능 완전 자동 구축

"사용자 인증 기능을 build me"

  1. autopilot 키워드 감지→자동 실행
  2. explore가 기존 코드 분석
  3. planner→executor→verifier

ralph: 복잡한 버그 끝까지 수정

"결제 모듈 500 에러, ralph로 고쳐줘"

  1. ralph 키워드 감지→실행
  2. debugger 가설 수립→executor 수정
  3. verifier 검증 실패 시 반복 루프

ultrawork: 대규모 병렬 리팩토링

"ulw — REST에서 tRPC 마이그레이션"

  1. 태스크를 파일/모듈 단위로 분해
  2. 3개 executor 병렬 실행
  3. 충돌 해결 후 전체 검증

ralplan: 아키텍처 설계 합의

"모노레포 vs 멀티레포 분석해줘"

  1. architect(Opus) 옵션별 장단점 분석
  2. critic 반론 및 엣지케이스 검토
  3. planner 최종 합의안 도출

"대시보드 만들고 싶은데 뭐가 필요한지 모르겠어"
