01

8가지 핵심 개념 한눈에 보기

Claude Code 생태계를 구성하는 개념들이다. 각각은 명확한 역할이 있지만, 실제 프로젝트에서는 서로 엮여서 동작한다.

S

Skill

Claude의 행동 방식을 바꾸는 지시 문서. 프롬프트에 주입되어 메인 Claude가 직접 따른다.
비유: 업무 매뉴얼 / SOP
T

Tool

파일 읽기, 편집, 검색, 명령 실행 등 Claude가 수행할 수 있는 원자적 행동 단위.
비유: 손에 쥔 도구 (망치, 드라이버)
A

Agent

별도 컨텍스트에서 독립 실행되는 Claude 인스턴스. 작업을 위임받아 자율적으로 수행한다.
비유: 일을 맡길 수 있는 팀원
M

MCP

외부 서비스(DB, API, Slack 등)를 연결하는 프로토콜. 외부 세계와의 인터페이스.
비유: USB 허브 / API 게이트웨이
Tm

Team

여러 Agent가 태스크 리스트를 공유하며 협업하는 조직 단위. 메시지로 소통한다.
비유: 프로젝트 팀 / 스크럼 보드
H

Hook

특정 이벤트(파일 저장, 세션 시작 등)에 자동으로 실행되는 명령. 강제 자동화.
비유: Git Hook / CI 파이프라인
P

Plugin

Skill + Agent + Hook + MCP를 묶어 배포하는 패키지. 설치 한 번으로 워크플로우 확장.
비유: VS Code Extension / npm 패키지
Me

Memory

대화 간 유지되는 영속 컨텍스트. 사용자 선호, 프로젝트 결정, 피드백을 기억한다.
비유: 팀 위키 / 회의록
02

개념별 상세

각 개념의 정의, 작동 방식, 파일 구조, 사용 시점을 정리한다.

SKILL

행동 지침 문서

Skill은 Claude의 행동 방식을 정의하는 마크다운 문서다. 프롬프트에 주입되어 메인 Claude가 직접 따른다. 코드가 아니라 지시문이다. "이런 상황에서 이렇게 일해라"는 매뉴얼에 해당한다.

파일 구조

  • SKILL.md — 스킬 본체 (YAML frontmatter + 마크다운)
  • supporting files — 참조 문서, 스크립트, 예시
  • 위치: plugin/skills/skill-name/

트리거 방식

  • 자동 트리거 — Claude가 상황 판단하여 호출
  • 슬래시 명령 — 사용자가 /skill-name으로 호출
  • 다른 스킬에서 호출 — 스킬 체이닝
# SKILL.md frontmatter 예시 --- name: brainstorming description: "Use when creating features or modifying behavior..." --- ## 동작 규칙 1. 명확화 질문을 한 번에 하나씩... 2. 설계 승인 없이 구현 불가 (Hard Gate)

핵심: Skill은 실행되는 코드가 아니라, Claude의 사고방식과 절차를 제어하는 지침이다. 메인 Claude 자체가 실행 주체.

TOOL

원자적 행동 단위

Tool은 Claude가 실제로 무언가를 수행할 때 사용하는 함수다. 파일을 읽고, 편집하고, 검색하고, 명령을 실행하는 등의 구체적 행동이다.

내장 Tool (항상 사용 가능)

  • Read — 파일 읽기 (이미지, PDF 포함)
  • Write — 파일 생성
  • Edit — 파일 수정 (diff 기반)
  • Glob — 파일 패턴 검색
  • Grep — 코드 내용 검색 (ripgrep)
  • Bash — 셸 명령 실행
  • Agent — 서브에이전트 생성
  • Skill — 스킬 호출

MCP Tool (외부 서비스 연동)

  • mcp__ide__* — IDE 연동
  • mcp__context7__* — 라이브러리 문서 조회
  • mcp__slack__* — Slack 메시지 (설치 시)
  • mcp__github__* — GitHub API (설치 시)
  • ... .mcp.json에서 구성

핵심: Tool = 실제 행동. Skill은 "어떻게 행동할지"를 알려주고, Tool은 그 행동을 실행한다.

AGENT

독립 실행 Claude 인스턴스

Agent는 별도의 컨텍스트 윈도우에서 실행되는 독립 Claude 인스턴스다. 메인 Claude와 대화 기록을 공유하지 않으며, 주어진 태스크만 집중적으로 처리한다.

왜 분리하는가?

  • 격리 — 구현자 ≠ 리뷰어 (편향 방지)
  • 컨텍스트 절약 — 대량 diff 분석을 서브에이전트가 처리
  • 병렬성 — 독립 태스크를 동시 실행
  • 집중 — 필요한 컨텍스트만 제공 → 정확도 향상

에이전트 유형

  • Explore — 읽기 전용 (Read, Grep, Glob)
  • Plan — 설계 전용 (읽기 + 계획)
  • general-purpose — 전체 Tool 접근
  • 커스텀 — agents/*.md로 정의
# agents/code-reviewer.md --- name: code-reviewer description: "Use after major project steps..." model: inherit --- ## Role 시니어 코드 리뷰어로서 계획 정합성, 코드 품질, 아키텍처를 검토한다. 이슈를 Critical / Important / Suggestion으로 분류한다.

Skill vs Agent 핵심 차이: Skill = "메인 Claude야, 이 규칙을 따라" / Agent = "새 Claude를 띄워서 이 일을 맡겨". Skill은 자기 행동 제어, Agent는 작업 위임.

MCP

외부 서비스 연결 프로토콜

MCP(Model Context Protocol)는 Claude가 외부 세계와 통신하는 표준 프로토콜이다. DB 쿼리, API 호출, Slack 메시지 전송 등을 가능하게 한다. 내장 Tool이 "로컬 파일 시스템"을 다룬다면, MCP는 "외부 서비스"를 다룬다.

서버 유형

  • Stdio — 로컬 프로세스 (가장 일반적)
  • SSE — 클라우드 서비스 (OAuth 지원)
  • HTTP — REST API 연동
  • WebSocket — 실시간 양방향

대표 MCP 서버

  • context7 — 라이브러리 최신 문서
  • Supabase — DB + Auth + Storage
  • GitHub — 이슈, PR, 워크플로우
  • Slack — 메시지 송수신
  • Sentry — 에러 트래킹
  • Playwright — 브라우저 자동화
// .mcp.json (프로젝트 루트에 위치) { "context7": { "command": "npx", "args": ["-y", "@upstash/context7-mcp"] }, "supabase": { "command": "npx", "args": ["supabase-mcp", "--project-id", "xxx"] } }

핵심: 내장 Tool = 로컬 파일 시스템, MCP Tool = 외부 서비스. MCP는 .mcp.json에 등록하면 Claude가 mcp__서버명__도구명으로 호출 가능.

TEAM

에이전트 협업 조직

Team은 여러 Agent를 하나의 조직으로 묶어 협업시키는 구조다. 공유 태스크 리스트로 작업을 조율하고, SendMessage로 소통한다. 단순 병렬 실행(dispatching-parallel-agents)과 달리 상태 공유와 의존성 관리가 가능하다.

구성 요소

  • config.json — 팀 멤버 정의
  • Task List — 공유 작업 목록 (pending → in_progress → completed)
  • SendMessage — 에이전트 간 메시지
  • blockedBy / blocks — 태스크 의존성

Agent와의 차이

  • Agent — 단일 작업 위임 (일회성)
  • Team — 다중 Agent 조율 (지속적)
  • Team은 태스크 큐 + 메시지 시스템을 제공
  • Agent는 결과만 반환하고 종료
HOOK

이벤트 기반 자동 실행

Hook은 특정 이벤트에 자동으로 트리거되는 명령이다. Skill이 "이렇게 해라"는 안내라면, Hook은 "무조건 실행되는 자동화"다. 선택이 아니라 강제.

이벤트 종류

  • SessionStart — 세션 시작/초기화
  • PreToolUse — Tool 실행 전 (차단 가능)
  • PostToolUse — Tool 실행 후
  • UserPromptSubmit — 사용자 입력 전처리
  • Stop — Claude 응답 종료
  • Notification — 알림 발생

실전 패턴

  • PostToolUse + Edit → 자동 포맷팅 (Prettier)
  • PostToolUse + Write → 자동 린팅 (ESLint)
  • PreToolUse + Bash → 위험 명령 차단
  • SessionStart → 스킬 시스템 초기화
  • Notification → 데스크탑 알림/소리
// settings.json (hooks 섹션) "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": "npx prettier --write $CLAUDE_FILE_PATH" }] }] }
PLUGIN

확장 패키지

Plugin은 Skill + Agent + Hook + MCP를 하나로 묶은 배포 단위다. 마켓플레이스에서 설치하면 내부 구성요소가 자동으로 등록된다. VS Code Extension이나 npm 패키지와 같은 개념.

포함 가능한 구성요소

  • skills/*/SKILL.md — 스킬들
  • agents/*.md — 커스텀 에이전트
  • hooks/hooks.json — 자동화 훅
  • .mcp.json — MCP 서버 설정
  • package.json — 메타데이터

관리 명령어

  • /plugin install name@marketplace
  • /plugin update name
  • /plugin list
  • /plugin info name
my-plugin/ ├── skills/ │ ├── brainstorming/SKILL.md │ └── debugging/SKILL.md ├── agents/ │ └── code-reviewer.md ├── hooks/ │ └── hooks.json ├── .mcp.json └── package.json
MEMORY

영속 컨텍스트

Memory는 대화가 끝나도 유지되는 기억 시스템이다. 사용자 프로필, 프로젝트 결정 사항, 피드백 등을 파일로 저장하여 다음 대화에서도 참조한다.

메모리 유형

  • user — 사용자 역할, 선호, 지식 수준
  • feedback — "이렇게 해라/하지마라" 피드백
  • project — 진행 중인 작업, 의사결정
  • reference — 외부 리소스 위치 포인터

파일 구조

  • MEMORY.md — 인덱스 파일 (항상 로드)
  • 개별 .md 파일 — 각 메모리 항목
  • YAML frontmatter (name, description, type)
  • 위치: ~/.claude/projects/{path}/memory/

비교표: 공통점과 차이점

8가지 개념을 7개 축으로 비교한다.

개념 본질 실행 주체 컨텍스트 영속성 수동/자동 정의 위치
Skill 지침 문서 메인 Claude 현재 대화에 주입 플러그인과 함께 자동 + 수동 skills/*/SKILL.md

공통점: 8개 개념 모두 "Claude의 능력을 확장"한다는 목적은 같다. 차이는 확장 방식이다 — 행동 지침(Skill), 행동 수단(Tool), 작업 위임(Agent), 외부 연결(MCP), 협업 구조(Team), 자동화(Hook), 패키징(Plugin), 기억(Memory).

05

실전 시나리오

SaaS 분석 대시보드 팀 개발

3인 팀이 SaaS 분석 대시보드를 개발한다. Next.js + Supabase 스택. 사용자 인증, 실시간 차트, 알림 시스템을 구축해야 한다. 이 프로젝트에서 8가지 개념이 어떻게 협력하는지 단계별로 추적한다.

Phase 0 — 환경 셋업

Plugin 설치 + MCP 구성 + Memory 초기화

superpowers 플러그인 설치 → 14개 스킬, code-reviewer 에이전트, 세션 훅 일괄 등록.
.mcp.json에 Supabase(DB), GitHub(PR/이슈), Slack(팀 알림), Sentry(에러 트래킹) MCP 서버 등록.
Memory에 팀 구성, 기술 스택 결정, 코딩 컨벤션 저장.

Plugin 설치 MCP 구성 Memory 초기화 SessionStart Hook 활성화
Phase 1 — 설계

"인증 시스템 + 대시보드 레이아웃을 만들자"

사용자 요청 → using-superpowers 스킬이 자동 판단 → brainstorming 스킬 트리거.
명확화 질문 (OAuth vs 이메일? RBAC 필요?) → 2~3개 접근법 제안 → 설계 문서 작성.
MCP로 Supabase 스키마를 조회하여 기존 테이블 구조 파악.
Memory에서 이전 대화의 기술 스택 결정 사항 참조.

brainstorming using-superpowers Supabase 스키마 조회 기술 스택 참조 Read, Grep (기존 코드 탐색)
Phase 2 — 계획

구현 계획서 작성

설계 승인 → writing-plans 스킬 자동 트리거.
파일 구조 매핑 → 2~5분 단위 태스크 18개로 분할.
각 태스크에 정확한 파일 경로, 완전한 코드, 테스트 명령, 예상 결과 포함.
using-git-worktrees 스킬로 격리된 작업 브랜치 생성.

writing-plans using-git-worktrees Write (계획서 저장) Bash (git worktree add)
Phase 3 — 구현

서브에이전트 기반 병렬 개발

자주 묻는 질문

Q. Skill도 결국 Agent 아닌가?

개념적으로는 맞다. 둘 다 "Claude의 능력을 확장"한다. 하지만 구현 관점에서 결정적 차이가 있다:
Skill = 메인 Claude의 행동을 바꾸는 프롬프트 주입 (같은 컨텍스트)
Agent = 새로운 Claude 인스턴스를 생성 (별도 컨텍스트)
Skill은 "네가 이렇게 해", Agent는 "다른 사람한테 맡겨".

Q. MCP Tool과 내장 Tool의 차이는?

내장 Tool = 로컬 파일 시스템과 상호작용 (Read, Write, Bash 등). 항상 사용 가능.
MCP Tool = 외부 서비스와 상호작용 (DB, API, Slack 등). .mcp.json에 등록해야 사용 가능.
내장 Tool이 "손", MCP Tool이 "전화기"라고 보면 된다.

Q. Hook과 Skill 둘 다 자동으로 실행되는데 뭐가 다른가?

Skill = Claude의 판단으로 트리거 ("이 상황에 이 스킬이 적합하다")
Hook = 이벤트 기반 무조건 트리거 ("Edit이 실행되면 무조건 Prettier 실행")
Skill은 유연한 판단, Hook은 기계적 강제 실행.

Q. Team과 dispatching-parallel-agents의 차이는?

dispatching-parallel-agents = 일회성 병렬 실행. 결과 받고 끝.
Team = 지속적 협업 구조. 태스크 큐, 의존성 관리, 메시지 소통이 가능.
간단한 병렬 작업은 dispatching, 복잡한 협업은 Team.

Q. Plugin 없이도 Skill을 사용할 수 있는가?

가능하다. CLAUDE.md에 직접 지시를 작성하거나, 프로젝트 내 .claude/skills/ 디렉토리에 SKILL.md를 넣으면 된다. Plugin은 "잘 만들어진 Skill 묶음을 쉽게 설치"하는 배포 수단일 뿐.

Q. Memory와 CLAUDE.md의 차이는?

CLAUDE.md = 사용자가 직접 작성하는 고정 지시문. 버전 관리됨. "이렇게 일해라".
Memory = Claude가 대화에서 학습하여 자동 저장. "이전에 이런 얘기를 했었다".
CLAUDE.md는 규칙, Memory는 기억.
